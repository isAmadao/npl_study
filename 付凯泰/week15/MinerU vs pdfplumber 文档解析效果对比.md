MinerU vs pdfplumber 文档解析效果对比
一、核心定位差异
pdfplumber
定位：轻量级 PDF 文本提取工具
技术路线：纯规则解析，基于 PyPDF2 底层
设计理念：快速提取可选文字 PDF 的原始文本流
MinerU
定位：智能文档理解系统（VLM + OCR + Pipeline）
技术路线：视觉语言模型 + 版面分析 + OCR 多模型协同
设计理念：像人类一样"读懂"文档结构和语义

二、关键能力对比

| 对比维度       | **MinerU**                                                   | **pdfplumber**                                               |
| -------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **技术本质**   | 多模型深度学习流水线（布局检测 + OCR + 公式识别 + 表格识别 + 阅读顺序恢复） | 基于 pdfminer.six 的规则式文本提取库，主要提供字符/线条几何信息 |
| **布局理解**   | ✅ **有**。LayoutLMv3/DocLayout-YOLO 检测 19 类布局元素，能区分正文、标题、页眉、页脚、侧边栏等 | ❌ **无**。仅按 PDF 内部内容流或简单坐标排序输出文本，不理解页面结构 |
| **阅读顺序**   | ✅ **显式恢复**。针对多栏、跨页、图文混排有专门的排序算法，OmniDocBench 上多栏布局的阅读顺序误差显著低于其他工具 | ❌ **无**。双栏文本会横向交错，导致语义断裂（实测结果）       |
| **表格处理**   | ✅ **专用模型**。TableMaster/RapidTable 识别表格结构，输出 Markdown/HTML 表格，支持合并单元格、跨页表格 | ⚠️ **有限**。`extract_tables()` 基于线条检测，对无框线表格、复杂合并单元格效果差，本次测试未识别出表格 |
| **数学公式**   | ✅ **检测+识别**。YOLOv8 先定位公式区域，UniMERNet 再转为 LaTeX，CDM 达 0.968 | ❌ **无**。公式被视为普通文本，行内公式常因字体编码问题出现乱码或丢失 |
| **页眉页脚**   | ✅ **自动过滤**。布局检测阶段即标记为 header/footer，不进入正文输出 | ❌ **无**。页眉、页脚、页码全部混入正文（实测 "Journal of Testing..." 出现在段首） |
| **输出格式**   | 结构化 Markdown / JSON，保留标题层级、表格、公式、图片引用   | 纯文本或简单的按行表格列表，无结构语义                       |
| **速度与资源** | 需要 GPU（推荐 CUDA），首次启动需加载数百 MB~GB 级模型，MinerU2.5 在 A100 上约 2.12 pages/s | 纯 CPU，毫秒级单页处理，资源占用极低                         |
| **适用场景**   | 学术论文、复杂财务报告、书籍、杂志等**版式复杂**的文档       | 简单单栏合同、发票、线条清晰的**简单表格**PDF                |
