# 作业2：06-stock-bi-agent项目分析

## 问题1：什么是前后端分离？

### 定义
**前后端分离**是一种软件架构设计模式，将前端（用户界面层）和后端（业务逻辑层、数据层）完全解耦，通过API接口进行通信。

### 在06-stock-bi-agent项目中的体现

#### 1. **后端（FastAPI服务）**
- **位置**: `main_server.py` + `routers/` + `services/` + `models/`
- **职责**:
  - 提供RESTful API接口
  - 处理业务逻辑（用户管理、聊天对话、股票数据等）
  - 数据库操作（SQLite ORM）
  - AI Agent集成（OpenAI Agents SDK）
  - MCP协议服务调用
  
- **技术栈**:
  ```python
  FastAPI          # Web框架
  SQLAlchemy       # ORM数据库操作
  OpenAI Agents    # AI智能体
  MCP Protocol     # 模型上下文协议
  ```

- **API示例**:
  ```python
  # routers/chat.py
  @router.post("/")
  async def chat(req: RequestForChat) -> StreamingResponse:
      # 处理聊天请求，返回流式响应
      return StreamingResponse(
          content=chat_stream_generator(),
          media_type="text/event-stream"
      )
  ```

#### 2. **前端（Streamlit应用）**
- **位置**: `demo/streamlit_demo.py` + `demo/chat/` + `demo/stock/` 等
- **职责**:
  - 用户界面展示
  - 用户交互处理
  - 调用后端API获取数据
  - 数据可视化（K线图、图表等）
  
- **技术栈**:
  ```python
  Streamlit        # 前端框架
  Requests         # HTTP客户端
  Plotly           # 数据可视化
  ```

- **API调用示例**:
  ```python
  # demo/chat/chat.py
  response = requests.post(
      "http://127.0.0.1:8000/v1/chat/",
      json={
          "content": prompt.text,
          "user_name": st.session_state['user_name'],
          "session_id": st.session_state['session_id'],
          "tools": selected_tool_names
      },
      stream=True  # 流式接收
  )
  ```

#### 3. **独立的MCP服务**
- **位置**: `main_mcp.py` + `api/`
- **职责**: 提供MCP协议工具服务
- **端口**: 8900
- **与主后端关系**: 被FastAPI服务通过HTTP调用

### 前后端分离的优势

| 优势 | 说明 | 项目中的体现 |
|------|------|-------------|
| **独立开发** | 前后端可以并行开发，互不干扰 | 后端提供API文档(`/docs`)，前端基于文档开发 |
| **技术栈灵活** | 前后端可以使用不同技术 | 后端用FastAPI，前端用Streamlit |
| **易于维护** | 模块化清晰，便于定位问题 | `routers/`处理路由，`services/`处理业务 |
| **可扩展性** | 可以轻松替换前端或后端 | 可以将Streamlit换成Vue/React，不影响后端 |
| **多端支持** | 一套API可服务多个前端 | 同一套API可服务Web、移动端、桌面端 |

## 问题2：历史对话如何存储，以及如何将历史对话作为大模型下一次输入

### 一、历史对话的存储机制

项目使用了**双层存储架构**：

#### 1. **关系型数据库存储（SQLite）**

##### 数据库文件
- **主数据库**: `./assert/sever.db`
- **对话数据库**: `./assert/conversations.db`

##### 数据表结构

**表1: `chat_session`（会话表）**
```python
# models/orm.py
class ChatSessionTable(Base):
    __tablename__ = 'chat_session'
    
    id = Column(Integer, primary_key=True, index=True)
```

```python
class ChatMessageTable(Base):
```

### 二、历史对话作为大模型下一次输入

#### 方式1: 通过AdvancedSQLiteSession自动管理（推荐）

这是项目中使用的主要方式，由OpenAI Agents SDK自动处理：

```python
# services/chat.py
async def chat(user_name: str, session_id: str, task: str, content: str, tools: List[str] = []):
    # 1. 创建或加载会话
    session = AdvancedSQLiteSession(
        session_id=session_id,
        db_path="./assert/conversations.db",
        create_tables=True
    )
    
    # 2. 初始化Agent
    agent = Agent(
        name="Assistant",
        instructions=get_init_message(task),  # System Prompt
        mcp_servers=[mcp_server],
        model=OpenAIChatCompletionsModel(
            model=os.environ["OPENAI_MODEL"],
            openai_client=external_client,
        ),
    )
    
    # 3. 运行对话（SDK自动加载历史对话）
    result = Runner.run_streamed(
        agent, 
        input=content,  # 当前用户输入
        session=session  # 传入session，SDK会自动加载历史
    )
    
    # SDK内部会：
    # - 从conversations.db加载该session_id的所有历史消息
    # - 将历史消息 + 当前输入一起发送给大模型
    # - 保持对话的连贯性
```

**优点**:
- ✅ 自动化管理，无需手动拼接历史
- ✅ 支持复杂的对话状态（包括工具调用）
- ✅ 由OpenAI官方SDK维护，稳定可靠

**缺点**:
- ❌ 依赖特定SDK
- ❌ 不够透明，难以自定义

#### 方式2: 手动从数据库读取并拼接（学习指南中提到）

在学习指南的第117行提到了这种实现方式：

> 自己实现：历史对话也输入给大模型，基于历史输入继续对话（限制最后10条历史对话作为输入，如果对话太长大模型忘记最开始信息）

**实现思路**:

```python
def get_chat_history_for_llm(session_id: str, max_messages: int = 10) -> List[Dict]:
    """获取最近N条历史对话，用于构建LLM上下文"""
    with SessionLocal() as session:
        # 查询该会话的所有消息
        chat_messages = session.query(ChatMessageTable) \
            .join(ChatSessionTable) \
            .filter(ChatSessionTable.session_id == session_id) \
            .order_by(ChatMessageTable.create_time.desc()) \
            .limit(max_messages) \
            .all()
        
        # 反转顺序（从旧到新）
        chat_messages = list(reversed(chat_messages))
        
        # 转换为LLM需要的格式
        messages = []
        for msg in chat_messages:
            if msg.role == "system":
                continue  # system message单独处理
            messages.append({
                "role": msg.role,
                "content": msg.content
            })
        
        return messages


async def chat_with_manual_history(user_name: str, session_id: str, content: str):
    # 1. 获取历史对话
    history_messages = get_chat_history_for_llm(session_id, max_messages=10)
    
    # 2. 构建完整的消息列表
    full_messages = [
        {"role": "system", "content": get_init_message(task)}
    ] + history_messages + [
        {"role": "user", "content": content}  # 当前输入
    ]
    
    # 3. 调用大模型
    response = await external_client.chat.completions.create(
        model=os.environ["OPENAI_MODEL"],
        messages=full_messages,
        stream=True
    )
    
    # 4. 流式返回并保存
    assistant_message = ""
    async for chunk in response:
        if chunk.choices[0].delta.content:
            yield chunk.choices[0].delta.content
            assistant_message += chunk.choices[0].delta.content
    
    append_message2db(session_id, "assistant", assistant_message)
```

**优点**:
- ✅ 完全可控，可以自定义逻辑
- ✅ 可以实现滑动窗口（只保留最近N条）
- ✅ 不依赖特定SDK

**缺点**:
- ❌ 需要手动管理消息拼接
- ❌ 不支持复杂的工具调用历史
- ❌ 需要自己处理token限制

### 五、优化建议

1. **限制历史长度**: 对于长对话，只保留最近N条消息，避免超出LLM的context window
2. **摘要压缩**: 对早期对话生成摘要，减少token消耗
3. **索引优化**: 为`session_id`和`create_time`添加数据库索引，提升查询性能
4. **异步优化**: 消息保存可以异步执行，不阻塞响应
5. **缓存机制**: 频繁访问的历史对话可以缓存到Redis

## 总结

### 问题1答案要点
- **前后端分离**是将UI层和业务逻辑层解耦的架构模式
- 本项目中：
  - 后端：FastAPI提供RESTful API（port 8000）
  - 前端：Streamlit负责界面展示（port 8501）
  - 通信：通过HTTP请求和SSE流式响应
- 优势：独立开发、技术灵活、易维护、可扩展

### 问题2答案要点
- **历史对话存储**采用双层架构：
  1. SQLite关系型数据库（`sever.db`）：存储结构化消息记录
  2. AdvancedSQLiteSession（`conversations.db`）：存储Agent对话状态
  
- **历史对话作为输入**有两种方式：
  1. **自动方式**（项目使用）：通过`AdvancedSQLiteSession`，SDK自动加载历史并拼接到LLM输入
  2. **手动方式**：从数据库查询历史消息，手动构建messages列表传给LLM
  
- **关键流程**：
  - 保存：`append_message2db()` + `Runner.run_streamed(session=session)`
  - 读取：`get_chat_sessions()`（前端展示）+ SDK自动加载（LLM输入）
  - 切换：通过`session_id`关联不同会话
