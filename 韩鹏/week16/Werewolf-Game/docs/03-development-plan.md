# AI狼人杀 — 敏捷开发计划

## 总览

| 阶段 | 名称 | 目标 | 可运行产物 |
|------|------|------|-----------|
| Phase 1 | 对局引擎 + 规则AI | 跑通一局完整狼人杀 | `python main.py` 终端输出完整对局 |
| Phase 2 | DeepSeek LLM Agent | LLM驱动的智能体对战 | `python main.py` 终端输出LLM对局 |
| Phase 3 | API服务 + 前端 | 网页实时观战 | 前后端联动的可视化对局 |
| Phase 4 | 集成 + 自演化 | 稳定运行 + 简单自演化 | 完整可部署系统 |

---

## Phase 1：对局引擎核心（1-2天）

### 开发目标
实现完整的狼人杀规则引擎，使用规则AI（非LLM）跑通一局完整对局。

### 开发内容

| 模块 | 文件 | 功能 |
|------|------|------|
| 角色定义 | `engine.py::Role` | 6个角色枚举 + 阵营归属 |
| 行动定义 | `engine.py::Action` | 查验/毒杀/投票/发言等行动类型 |
| 规则引擎 | `engine.py::RuleEngine` | 行动合法性检查 + 行动结算 |
| 状态机 | `engine.py::StateMachine` | 回合流转：夜晚→白天→投票→夜晚 |
| 规则AI | `engine.py::RuleAgent` | 启发式策略：狼人随机杀人、预言家随机查验等 |
| 主入口 | `main.py` | 终端运行，打印完整对局日志 |

### 游戏流程（6人局）
1. 分配角色：2狼人 + 1预言家 + 1女巫 + 2村民
2. 夜晚：狼人选目标 → 预言家查验 → 女巫用药/毒
3. 天亮：公布死亡结果
4. 白天发言：每个存活玩家发言
5. 投票放逐：最高票出局，公布身份
6. 循环直到：狼人全灭（好人胜）或 狼人数≥好人数（狼人胜）

### 测试目标
- 单元测试：角色分配、行动合法性、投票计数、胜负判定
- 集成测试：模拟完整对局，验证状态机正确流转
- 终端运行 `python main.py` 输出完整日志

### 验收标准
- [x] 终端运行输出完整对局过程
- [x] 行动合法性检查正确
- [x] 胜负判定正确
- [x] 测试全部通过

## Phase 2：DeepSeek LLM Agent（2-3天）

### 开发目标
接入DeepSeek v4-flash API，将规则AI替换为LLM Agent，实现真正智能体对战。

### 开发内容

| 模块 | 文件 | 功能 |
|------|------|------|
| LLM客户端 | `agent.py::LLMClient` | 封装DeepSeek API调用，处理重试/超时 |
| Prompt模板 | `agent.py::PROMPTS` | 每个角色的System Prompt + 行动Prompt |
| 信息过滤器 | `agent.py::InfoFilter` | 从完整状态提取Agent可见信息 |
| LLM Agent | `agent.py::LLMAgent` | 构造Prompt → 调用LLM → 解析行动 |

### Prompt设计要点
- System Prompt：角色身份、胜利条件、可用行动、信息边界
- 行动Prompt：当前游戏状态 + 可见信息 + 要求输出JSON
- 发言Prompt：辩论上下文 + 身份伪装建议（狼人）

### 测试目标
- 单元测试：InfoFilter正确性、Prompt格式、JSON解析
- API连通性测试：调用DeepSeek获取有效响应
- 集成测试：至少跑3局完整LLM对局，验证无异常

### 验收标准
- [ ] LLM Agent正确输出结构化JSON行动
- [ ] 信息过滤正确（预言家看不到狼人身份等）
- [ ] 完整对局无阻塞/死循环
- [ ] 每局输出结构化日志

## Phase 3：API服务 + 前端UI（2-3天）

### 开发目标
搭建FastAPI + WebSocket后端，Vue3前端实时观战。

### 后端开发内容

| 模块 | 文件 | 功能 |
|------|------|------|
| FastAPI应用 | `server.py` | HTTP路由 + WebSocket端点 |
| 游戏管理 | `server.py::GameManager` | 创建/查询游戏，管理生命周期 |
| WebSocket推送 | `server.py::ws_endpoint` | 游戏事件实时推送 |

### 前端开发内容

| 组件 | 文件 | 功能 |
|------|------|------|
| 主布局 | `App.vue` | 页面框架 |
| 玩家面板 | `GameBoard.vue` | 存活/死亡状态、身份标记 |
| 日志流 | `ChatLog.vue` | 实时发言/行动日志滚动 |
| 控制面板 | `GameControl.vue` | 启动新游戏、速度调节 |

### API设计
- `POST /api/games` — 创建新游戏，返回game_id
- `GET /api/games/{id}` — 查询游戏状态
- `WS /ws/games/{id}` — WebSocket，推送事件流

### 测试目标
- 后端API测试（HTTP endpoint + WebSocket连接）
- 前端组件渲染测试
- 端到端：点"新游戏" → 前端实时显示对局全过程

### 验收标准
- [ ] 前端页面可访问
- [ ] WebSocket连接正常
- [ ] 实时显示对局过程（发言、投票、死亡等）
- [ ] 新游戏按钮可正常触发

## Phase 4：集成测试 + 自演化（1-2天）

### 开发目标
端到端整合，添加简化版自演化记忆系统，确保系统稳定。

### 开发内容

| 模块 | 功能 |
|------|------|
| 对局记忆存储 | 每局结束后保存Agent的关键决策和结果到JSON |
| 经验注入 | 下一局Agent的System Prompt追加历史经验摘要 |
| 一键启动 | `start.bat` / `start.sh` 启动全部服务 |
| 集成测试 | 端到端自动化测试：创建游戏→完整对局→前端接收 |

### 验收标准
- [ ] 自演化：第N局Agent能看到前N-1局的经验
- [ ] 一键启动脚本正常运行
- [ ] 连续3局无中断
- [ ] 全部测试通过
