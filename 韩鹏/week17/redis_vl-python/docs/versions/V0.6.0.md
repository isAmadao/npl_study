# V0.6.0 — 集成联调与 E2E 测试

## 版本目标

## 手动测试详细流程

以下流程已按当前本地环境验证过，端口以仓库 `.env` 中配置为准：

1. 启动 Redis Stack。
   - 推荐命令：`docker compose up -d`
   - 当前仓库环境通常使用 `REDIS_PORT=16379`
2. 激活 Python 环境。
   - 命令：`conda activate py310`
3. 启动应用。
   - 命令：`python run.py`
   - 默认访问地址：`http://localhost:18000`
4. 打开首页。
   - 检查页面是否出现：
     - `V0.6.0`
     - “清除语义缓存”按钮
     - “最近 5 轮会话时间线”区域
5. 准备一个全新的 Session ID。
   - 示例：`manual-v06-clean`
6. 在该 Session 下首次输入：`Python 列表怎么排序`
   - 预期：
     - `EmbeddingsCache`：未命中
     - `SemanticCache`：未命中
     - `SemanticRouter`：匹配 `技术问答`
     - `SemanticMessageHistory`：`count = 0`，`recent_count = 1`
     - 顶部显示响应耗时和调用顺序
7. 再次输入同样的问题：`Python 列表怎么排序`
   - 预期：
     - `EmbeddingsCache`：命中
     - `SemanticCache`：命中
     - `SemanticMessageHistory`：`count = 1`，`recent_count = 2`
     - 时间线出现两条记录
8. 输入闲聊问题：`你好啊今天怎么样`
   - 预期：
     - `SemanticRouter`：匹配 `日常闲聊`
9. 输入数学问题：`3.14 乘以 2`
   - 预期：
     - `SemanticRouter`：匹配 `数学计算`
10. 切换到另一个 Session ID。
    - 示例：`manual-v06-another`
    - 页面应自动刷新时间线
    - 预期：
      - 时间线为空或只显示该 session 自己的历史
      - 不会混入前一个 session 的记录
11. 点击“清除当前历史”。
    - 预期：
      - 当前 session 时间线立即清空
      - `MessageHistory` 卡片显示已清除提示
12. 点击“清除语义缓存”。
    - 预期：
      - 页面状态提示显示清除成功
      - 下一次同样的问题会重新变成 `SemanticCache` 未命中
13. 可选边界验证。
    - 输入空白问题：页面应提示“请输入问题后再发送”
    - 使用特殊字符 Session ID：例如 `特殊 / session ? id`
    - 预期：
      - 接口仍能正常工作
      - 后端会返回规范化后的 `normalized_session_id`
